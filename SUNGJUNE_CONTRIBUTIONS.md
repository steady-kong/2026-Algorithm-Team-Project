# Sungjune 기여 정리

팀원 리뷰·머지를 위해, 내가 구현한 기능들의 **목적·원리·변경 파일**을 정리한다.
이 브랜치(`Sungjune`)의 **신규 변경은 "추천 근거(why) 표시" 하나**이고, 나머지는 이미
`main`에 머지된 기여라 맥락용으로 아래에 요약해 둔다.

---

## 1. 추천 근거(why) 표시 — 이 브랜치의 신규 기능

### 목적
추천 카드마다 **"왜 이 메뉴인지"** 를 한 줄로 보여 준다. "LLM 은 후보를 생성하고,
직접 구현한 알고리즘이 점수로 고른다" 는 프로젝트 컨셉을 사용자에게 드러내는 게 핵심이다.
LLM 이 만든 마케팅 문구(`tagline`)와 달리, 이건 **점수 기반의 결정적(算法) 설명**이다.

### 원리
- 서버가 후보를 랭킹할 때 쓰는 `cupMatchScore`(산미·바디·단맛·쓴맛 4축, 각 축 `1 - |예상−취향|/4`
  의 평균)와 **같은 기준**으로 근거를 만든다.
- **사용자가 메시지에서 실제로 언급한 맛 축만** 근거로 삼는다. 추론된 프로파일이 아니라
  **발화(메시지) 기준**이라, 언급도 안 한 축을 "취향에 맞다" 고 꾸며 말하지 않는다.
  (정규식으로 산미/바디/단맛/쓴맛 언급을 보수적으로 감지)
- 언급한 축 가운데 예상치와 **잘 맞는(차이 ≤ 1) 축만** 짚고 적합도% 를 덧붙인다.
  언급한 축이 없거나 잘 맞는 축이 없으면 **빈 문자열** → 카드에서 근거 줄 자체를 숨긴다.
- 한국어 조사(이/가)는 끝 글자 받침으로 자동 선택한다.

예: `🎯 단맛이 취향에 잘 맞아요 · 적합도 100%` / 언급 없는 요청엔 근거 줄 없음.

### 설계 결정(왜 발화 기준인가)
초기 구현은 "예상치와 가까운 축은 전부" 표기했는데, 취향 정보가 적으면 프로파일이
중립(3)에 가까워 **사용자가 말한 적 없는 산미·바디까지 "맞다"** 고 나왔다. 근거 없는
사유는 신뢰를 해치므로 → **사용자가 실제 말한 축에만 한정**하고, 없으면 숨기도록 바꿨다.

### 변경 파일
| 파일 | 내용 |
|------|------|
| `website/src/lib/util/match-explain.ts` (신규) | `explainCupMatch(cup, profile, message, locale)` — 발화 기준 근거 생성 |
| `website/src/lib/types/proposal.ts` | `Proposal.why?` 필드 추가 |
| `website/src/routes/api/chat/propose/+server.ts` | 각 추천에 `why` 부착 |
| `website/src/routes/api/chat/refine/+server.ts` | mod/alt/일반 추천에 `why` 부착 |
| `website/src/lib/components/ProposalCards.svelte` | 🎯 근거 줄 렌더(`why` 있을 때만) |

### 검증
- `cd website && npm run check` → 0 errors.
- 로컬 실측: "달콤한 거" → `단맛`, "산미 있는 산뜻한 거" → `산미`, 맛 축 언급 없으면 줄 숨김.

## 2. 이미 `main`에 머지된 기여 (PR #6, 맥락용 요약)

> 아래는 이미 `main`에 반영되어 있고(팀원이 일부 추가 개선함), 이 브랜치에 다시 담지 않는다.

- **콜드브루 명시-요청 게이트**: 콜드브루는 침지식이라 드립·에스프레소 변주와 본질적으로
  다르다. 사용자가 명시적으로 요청했을 때만 추천 후보에 포함하도록 propose/refine 에 게이트 추가.
- **비정상(off-topic) 요청 거절 게이트**: 추천 루프 전에 "커피·음료 요청인가?" 를 분류해,
  무관(코딩·수학·잡담 등)하면 아무 레시피도 만들지 않고 정중히 거절. (이후 팀원이 도메인
  키워드 결정적 1차 통과를 더해 비결정성을 보강함)
- **about 플로우차트 직관화**: 떠다니던 삼각형 글리프를 실선 연결선 + 화살촉으로 바꾸고,
  분기는 "계속 = 중앙 레일 직진 / 타깃 분기 = 옆으로 빠져 종료" 로 재구성.
